# 排课算法简要说明

当前项目的核心排课算法位于 `src/sjtu_course_analysis/scheduler.py`，目标是在用户给定的必修课和可选课中，选出一组时间不冲突、评分尽可能高的教学班组合。

## 输入与候选集

用户输入分为两类课程：

- `compulsory`：必选课程，必须尽量全部排入课表。
- `optional`：可选课程，可以选择排入，也可以跳过。

算法会从 SQLite 数据库中读取这些课程的所有教学班，并关联课程评价汇总表，得到每个教学班的：课程号、课程名、教师、教学班、学分、上课时间、平均评分和评价数。

如果某个教学班没有评分，会使用默认分数 `unrated_score`。如果课程时间无法解析，或者学分无效，该教学班会被跳过并记录 warning。

## 时间表示

数据库中的 `schedule_code` 会被解析成离散的时间槽集合 `Slot`：

```text
Slot = (week, day, period)
```

例如：

```text
D1:P1-2:W1-8
```

表示第 1 到 8 周、星期一、第 1 到 2 节。算法把它展开成多个具体的 `(周次, 星期, 节次)`，后续判断冲突时只需要检查两个教学班的时间槽集合是否有交集。

## 候选压缩

同一门课可能有多个教学班时间完全相同。为了减少搜索规模，算法会按：

```text
(时间槽集合, 学分)
```

对教学班分组。对于时间和学分完全相同的教学班，只保留排序最优的一个。排序依据依次是：

1. 平均评分更高；
2. 评价数更多；
3. 学分更高；
4. 教学班编号字典序更大。

这样可以在不影响最优课表结果的前提下减少候选数量。

## 搜索过程

核心搜索使用深度优先搜索（DFS）。算法会先把课程按候选教学班数量从少到多排序，这样更早处理约束更强的课程，减少无效分支。

搜索时维护以下状态：

- 已占用时间槽 `occupied`；
- 已选择的教学班 `selected`；
- 当前评分加权和 `weighted_sum`；
- 当前总学分 `total_credits`；
- 当前早八课程数量 `early_class_count`。

对于必修课，算法尝试选择一个不与现有课表冲突的教学班。对于可选课，算法会先尝试“跳过该课程”，再尝试选择其中一个不冲突的教学班。

如果设置了 `max_early_classes`，当当前早八数量超过限制时，该搜索分支会被剪枝。

当启用 `approx` 模式时，系统改用 beam search 近似搜索：每处理一门课，只保留当前评分排序最高的前 `beam_width` 个部分课表，并且每门课最多尝试前 `per_course_limit` 个候选教学班。该模式仍会检查时间冲突和早八限制，能快速返回可行课表，但不保证全局最优。为提高扩展速度，beam search 内部使用整数位图表示已占用时间槽。

## 最优目标

每个可行课表都会计算加权平均评分：

```text
weighted_average = sum(教学班评分 * 学分) / sum(学分)
```

算法比较课表优劣时使用以下优先级：

1. 加权平均评分更高；
2. 选入的可选课数量更多；
3. 总学分更高；
4. 评分加权和更高；
5. 早八课程数量更少。

最终返回优先级最高的课表。如果不存在满足条件的组合，则返回 `infeasible`。

## 输出结果

算法输出 `TimetableResult`，主要包含：

- `status`：`optimal` 或 `infeasible`；
- 实际选中的教学班列表；
- 选中和跳过的可选课；
- 缺失的必修课和可选课；
- 总学分、评分加权和、加权平均评分；
- 早八课程数量；
- 搜索统计信息，例如原始候选数、压缩后候选数、访问节点数；
- warnings。

整体来说，这个算法是一个带候选压缩和简单剪枝的穷举搜索算法，适合在课程数量和候选教学班数量中等的情况下找到全局最优课表。
